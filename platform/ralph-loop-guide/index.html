什么是 Ralph Loop

核心概念

Ralph Loop 是一种基于 Claude Code Stop Hook 的迭代开发方法论，由 Geoffrey Huntley 首创。其本质是一个 while true 循环 — 反复将同一个 Prompt 喂给 AI，让它通过读取文件中的上一轮产出来持续改进工作。

# 传统 bash 实现
while :; do
  cat PROMPT.md | claude-code --continue
done

# Claude Code Plugin 实现 (Stop Hook 自动循环)
/ralph-loop "你的任务描述" --completion-promise "DONE"

工作原理

01
接收 Prompt
Claude 收到同一份 Prompt
02
执行任务
修改文件、运行测试
03
Stop Hook 拦截
退出被拦截并重启
04
迭代改进
读取上轮产出继续优化
自引用机制：Prompt 永远不变，但每一轮 Claude 都能看到上一轮写入文件的修改和 git 历史，从而实现增量改进。循环持续到输出 <promise>DONE</promise> 或达到最大迭代次数。

适用场景

适合
  • 有明确成功标准的任务
  • 需要迭代和自我纠错的开发
  • 可自动验证的任务（测试、编译、E2E）
  • 绿地项目 — 可以挂机运行
不适合
  • 需要人工判断的设计决策
  • 一次性操作（部署、数据迁移）
  • 成功标准模糊的任务
  • 生产环境调试

每轮 4 阶段流程

所有 5 个模块共享相同的 4 阶段结构，每一轮迭代都按顺序执行这 4 个阶段：

Phase 1 — 测试 / 检查

使用自动化工具执行检查，收集原始数据。

模块Phase 1 内容工具
SmartBI 全模块Playwright E2E 遍历 11 个页面 + 交互测试 + 截图Playwright MCP
RN 移动端Expo Web 登录 → 全 Tab 遍历 → 交互测试Playwright MCP
AI 意图mvn compile + Tool 架构审查Maven + 代码审查
Python 服务语法检查 + API curl 测试py_compile + curl
Web Admin全页面菜单遍历 + 表格/表单测试Playwright MCP

Phase 2 — 分析

使用 /agent-team 多角色深度评估测试结果，输出按优先级排序的问题清单。

P0 功能阻断 P1 体验问题 P2 美化空间

评估维度包括：截图对比、上轮回归检查、代码规范合规、竞品差距分析。

Phase 3 — 修复

按 P0 → P1 → P2 优先级修复问题。每轮最多修 3-5 个文件，避免改动过大引入新 bug。修复后必须运行验证命令。

# 修复后验证命令
cd backend/java/cretas-api && mvn compile -q       # AI 意图
cd frontend/CretasFoodTrace && npx tsc --noEmit     # RN 移动端
cd web-admin && npx vite build                       # Web Admin / 财务看板
cd backend/python && python -m py_compile main.py    # Python 服务

Phase 4 — 调研 + 下轮计划

使用 /agent-team 进行竞品调研和行业最佳实践研究，输出下一轮的测试重点和优化方向。

完成条件：当连续一轮全部测试通过且无新问题时，输出 <promise>MODULE PERFECT</promise>，循环自动停止。

启动命令

启动 Ralph Loop

在 Claude Code 中运行以下命令，将 Prompt 文件喂入循环：

# 基本语法
/ralph-loop "$(cat .claude/ralph-prompts/MODULE.md)" \
  --completion-promise "PROMISE_TEXT" \
  --max-iterations 30
模块启动命令完成 Promise
SmartBI 全模块 /ralph-loop "$(cat .claude/ralph-prompts/financial-dashboard.md)" --completion-promise "SMARTBI PERFECT" SMARTBI PERFECT
RN 移动端 /ralph-loop "$(cat .claude/ralph-prompts/rn-app.md)" --completion-promise "RN APP PERFECT" RN APP PERFECT
AI 意图 /ralph-loop "$(cat .claude/ralph-prompts/ai-intent.md)" --completion-promise "INTENT CLEAN" INTENT CLEAN
Python 服务 /ralph-loop "$(cat .claude/ralph-prompts/python-services.md)" --completion-promise "PYTHON CLEAN" PYTHON CLEAN
Web Admin /ralph-loop "$(cat .claude/ralph-prompts/web-admin.md)" --completion-promise "WEB ADMIN PERFECT" WEB ADMIN PERFECT

取消循环

# 取消当前运行的 Ralph Loop
/cancel-ralph

也可以直接按 Ctrl+C 终止进程，但推荐使用 /cancel-ralph 来清理状态文件。

参数说明

参数说明默认值
--max-iterations <n>最大迭代次数，超过自动停止无限制
--completion-promise <text>完成信号文本，精确匹配无
安全提示：始终设置 --max-iterations 作为安全网，防止在不可能完成的任务上无限循环消耗 API 额度。

自定义 Prompt

Prompt 编写规范

创建新的 Ralph Loop Prompt 时，遵循统一的 4 阶段结构：

# [模块名] — Ralph Loop [测试类型]循环指令

你是 [角色描述]。按照以下 4 阶段执行本轮迭代。

---

## Phase 1: 测试 / 检查
# 使用工具执行自动化检测，收集数据

## Phase 2: 分析
# 使用 /agent-team 多角色评估
# 输出 P0/P1/P2 问题清单

## Phase 3: 修复
# 按优先级修复，每轮限制修改文件数
# 修复后运行验证命令

## Phase 4: 调研 + 下轮计划
# 使用 /agent-team 进行竞品调研
# 输出下轮测试重点

---

## 完成条件
当 [条件] 时输出: <promise>PROMISE_TEXT</promise>

Prompt 最佳实践

BAD

"把代码写好，修所有 bug。"

  • 没有明确的完成条件
  • 没有阶段划分
  • 没有修复数量限制
GOOD

"E2E 测试所有页面 → 按 P0-P2 分析 → 每轮最多修 5 个文件 → 输出下轮计划。全部通过时输出 <promise>"

  • 明确的成功标准
  • 增量式阶段
  • 自我纠错机制

临时运行（不创建文件）

如果只是临时测试，可以直接在命令中写 Prompt，不需要创建 .md 文件：

/ralph-loop "修复 auth.ts 中的 token 刷新逻辑。
运行 npx tsc --noEmit 确认编译通过。
全部修复后输出: <promise>FIXED</promise>" \
  --completion-promise "FIXED" \
  --max-iterations 10
✓

最佳实践

4 大原则

  1. 迭代 > 完美 — 不要追求首轮完美，让循环自然迭代改进
  2. 失败是数据 — "确定性地坏"意味着失败是可预测的、有信息量的，用失败来调优 Prompt
  3. Operator Skill 很关键 — 成功取决于写好 Prompt，而不只是模型能力
  4. 坚持就是胜利 — 循环自动处理重试逻辑，只要 Prompt 对了就能收敛

实战技巧

  • 每轮限制修改文件数（3-5 个），避免改动过大引入回归
  • Phase 2 分析时使用 /agent-team 多角色评估，不要只用单一视角
  • Phase 4 调研时开启 BROWSER_RESEARCH=true，让 agent 实际浏览竞品页面
  • 为编译型项目（Java/TypeScript），每轮修复后必须运行编译验证
  • 首次运行建议设 --max-iterations 10，观察收敛情况后再调整
⚠

注意事项

API 成本：Ralph Loop 会持续消耗 Claude API 额度。30 轮迭代约消耗 $15-50（取决于任务复杂度和上下文长度）。务必设置 --max-iterations 上限。
Completion Promise 是精确匹配：Claude 只能通过输出 完全匹配 的 <promise> 标签来退出循环。不要设置多个退出条件（如 "SUCCESS" vs "BLOCKED"），始终依赖 --max-iterations 作为安全机制。
状态文件：Ralph Loop 在运行时创建 .claude/.ralph-loop.local.md 状态文件。如果 Claude Code 异常退出，手动删除此文件即可清理残留状态。

常见问题

问题解决方案
循环不停止检查 Promise 文本是否精确匹配，或使用 /cancel-ralph 手动停止
每轮重复相同操作Prompt 中缺少"检查上轮修复"步骤，添加 Phase 2 回归检查
修改引入更多 bug减少每轮修改文件数，增加验证步骤
E2E 测试连不上目标确认目标服务已启动，检查端口和 URL
异常退出后残留状态手动删除 .claude/.ralph-loop.local.md
