这篇文章（你截图里关于“SAP模式救不了用友”“AI把厚重压成对话”“最后10%才是命根子”）本质上在讲一件事：**中国企业软件的竞争，已经从“功能堆砌/咨询驱动”切换为“交付效率/可配置/AI助推的产品化交付”**。对我们白垩纪来说，它是一个很清晰的“避坑清单 + 发力方向”。

## 一、这个信息说明什么（对中国工厂软件的“现实主义总结”）

### 1) “SAP式”打法在中国工厂现场容易失灵：不是功能不强，而是**交付逻辑不匹配**

文章强调的冲突点是：

* SAP/传统大厂常用“最佳实践蓝图 → 流程变革咨询 → 再上系统”的路径；
* 中国老板更常见的路径是“先跑起来拿数据 → 边用边改 → 能立刻出报表/出结果”。
  所以客户会问“能不能一行代码不改就接住？”背后是：**客户不愿意为大改造买单，更不愿意为长期项目承担组织成本**。

### 2) 护城河从“厚重功能”转向“交付速度 + 最后10%适配能力”

你截图里那句“90%不重要，10%才是命根子”，意思是：

* 90%通用流程大家都能做；
* 真正决定成败的是：**能否用很小的成本，把客户那点“特殊但关键”的差异快速接住**（且不把产品做烂）。

### 3) AI正在把“使用门槛”从表单/菜单压缩成对话，但AI本身不会自动带来交付效率

文章也提到“把厚重一键压缩成对话”。这对用户体验是利好，但对厂商是压力：

* 大模型让“生成报表/解释数据/写口径”变得更便宜；
* 如果你的系统底座数据不完整、口径不一致、流程不闭环，再强的AI也只能“讲故事”。
  结论：**AI是放大器，不是地基。**

### 4) 中国市场正在惩罚“大客户依赖 + 高交付成本”模式

大客户一收预算，厂商就“被拔氧气管”。对创业公司尤其关键：

* 不能把公司绑定在一两个大客户的深度定制上；
* 必须把“交付”产品化，把“实施成本”压到可规模化的水平。

## 二、白垩纪需要注意什么（用这篇文章当“路线图校正器”）

你们的PRD里已经把方向定得很对：**多租户、移动优先、精细权限、加工/质检/库存/成本、再叠加AI分析**（这与“现场能用、快速落地”的逻辑一致）。
真正要注意的是：不要把自己做成“另一个用友/另一个MES”，而要在**交付机制**上做出代差。

### A. 产品层：把“定制化”变成“配置化 + 插件化”，核心代码尽量不动

**要点：客户要的是“像定制一样贴合”，但你要用“配置/模板”交付。**

建议你们把所有需求拆成三层：

1. **标准层（80%）**：食品加工的共性闭环（入库→生产批次→消耗→质检→成品→成本/良率）。这是你们的“开箱即用”。（你们现有加工、库存、成本、质检方向已经在路上）
2. **行业参数层（15%）**：字段、表单、质检项、批次状态、成本口径、报表维度，用“配置中心”解决。
3. **最后10%（5%）**：客户差异化最强的那一点，用**插件/规则引擎/脚本扩展位**解决，而不是每次改主干代码。

> 你们现在的风险点是：如果客户一提“我们27家子公司都不一样”，你们被迫走“项目制深定制”，就会走到文章批评的那条路上。

### B. 交付层：必须把“实施”做成可复制的产品，而不是靠人堆

文章里对“驻场工程师带睡袋”的隐喻很典型。白垩纪要反过来：

* **交付包标准化**：1周上线、2周跑通闭环、4周出第一版降本结论（形成你们自己的SOP）。
* **数据采集最小化**：不要让客户填100个字段才开始有价值；你们移动端的优势要转化成“极少步骤完成关键记录”。
* **把报表/口径固化**：客户真正买单往往是“凌晨两点把那张报表改到他点头”为止。你们要让这件事变成“改配置/改口径版本”，不是改代码。

### C. AI层：AI要为“交付效率”服务，而不是只做“分析展示”

你们文档里AI主要围绕成本、效率、质量、预测等分析输出。
建议明确两类AI能力，优先级不同：

1. **P0：AI做“数据闭环助手”**（直接减少实施成本）

* 对话式收集缺失字段、自动补全表单、提示异常口径、生成质检整改任务；
* 让AI把“现场不会填/填不齐”的问题吞掉，而不是事后分析“数据缺失”。

2. **P1：AI做“管理层可决策的解释与建议”**（增强续费与扩张）

* 你们已有的成本分析/效率评估/质量分析/预测输出就是这一类。

一句话：**AI优先把“交付的人工”打下去，而不是先把“PPT感的智能”做上去。**

### D. 商业与客户策略：别被“大客户深定制”带偏

这篇文章的“警报”对你们尤其重要：

* 你们应该以“中小/中型食品工厂的可复制场景”为主战场，把“能快速跑通闭环”做成品牌；
* 大客户可以做，但必须坚持“产品化边界”，否则会把团队拖进项目泥潭，实施成本上升，迭代速度下降。

### E. 你们对外表达要升级：不要说“我们比ERP/MES强”，要说“我们把交付做成了产品”

面向质疑（“你们不就是ERP/MES吗？”）时，用一套更“现场主义”的话术：

* 传统ERP/MES：**先定义流程→再让人按流程填→最后汇总报表**
* 白垩纪：**先把现场关键事件采集起来→系统自动形成批次链路与成本口径→AI把异常和动作直接推到人**
  核心不是功能名，而是**数据形成方式 + 落地速度 + 后续迭代方式**。

## 三、给白垩纪的“硬指标”（用来约束产品不走歪）

建议你们内部用三条KPI约束路线（和文章表达的方向一致）：

1. **Time-to-Value**：从签约到“第一张老板认可的成本/良率报表”≤ 14天
2. **配置覆盖率**：客户提出的变更，≥80%可通过配置/规则完成（不改主干代码）
3. **交付成本率**：每新增一个客户，实施人天随客户规模线性增长的部分要越来越小（交付可规模化）

  并且把“最后10%”常见需求（食品工厂最典型的20类差异）整理成一套可复用模板库。

  1）哪些页面/字段必须做成可配置（否则一定走向“改代码交付”）

原则：凡是“工厂差异最大、且经常被客户要求改”的，必须配置化；凡是“核心账本一致性要求高”的，尽量标准化。

1.1 必须配置化的页面类型（按优先级）

P0：最容易被定制的“业务记录页”

原料入库批次（到货/入库/供应商批号/到货重量/抽检结论）

生产批次（投料/工序/线别班组/产出/损耗）

质检记录（来料/过程/成品；抽检规则；判定阈值）

成品入库/出库批次（批次关联、去向、召回关联字段）

P1：容易“每个厂都不一样”的“异常与整改页”
5) 异常上报（异物、超差、短重、温控、标签等）
6) 返工/报废/让步放行（处置路径、审批人、留痕字段）
7) 追溯/召回任务（影响范围、召回等级、对外口径字段）

P2：管理侧“看数口径”相关页面
8) 报表口径设置（损耗是否入成本、辅料摊销、边角料回收等）
9) 多维筛选与维度管理（线别/班组/供应商/品类/规格/客户）

1.2 在这些页面里，哪些字段要“配置化能力”支撑（能力清单）

你要做的不是“能加字段”，而是一套完整的表单元数据能力，至少包括：

字段级配置

自定义字段（文本/数字/枚举/日期/附件/图片/扫码）

字段别名（同一字段不同叫法：如“投料重量/投料净重”）

必填规则（按产品/工序/角色/状态动态必填）

显示规则（条件展示：例如“只有不合格才出现不合格原因”）

默认值/自动填充（班组、线别、时间、上一次选择）

单据级配置

单据模板（不同产品线/不同车间用不同模板）

编码规则（批次号/托盘号/箱码：前缀+日期+流水+校验位）

权限矩阵（谁能看、谁能改、谁能审批、谁能导出）

状态机（“待检→合格→入库/不合格→复检→处置”等可配置）

质检项配置（非常关键）

质检项目库（项目、单位、阈值、判定方式）

抽检策略（按批次比例/按时间/按供应商风险等级）

判定规则（任一超差即不合格 / 加权评分 / 允许让步阈值）

交付层面的硬要求：实施人员在后台“点点选选”就能改这些，而不是提需求给研发。

2）哪些地方要预留插件位/规则引擎（把“最后10%”变成可控扩展，而不是改主干）

你需要两类扩展机制：规则（低代码、可配置、可审计） + 插件（代码级、隔离、可热插拔）。

2.1 规则引擎：适合“经常变、逻辑不复杂、必须可审计”的差异

建议你把规则引擎做成“事件触发 + 条件 + 动作”的模式（IF/THEN），覆盖这些点：

A. 表单与校验类

提交前校验：重量区间、必填、附件要求、重复批次检查

自动补全：根据产品/工序自动带出默认质检项、阈值、损耗原因选项

B. 流程与权限类

状态流转条件：例如“不合格必须走复检或处置”

审批链路：按金额/风险等级/客户类型路由不同审批人

C. 计算口径类

成本摊销规则（辅料、包材、人工、能耗按何种分摊）

损耗归因与统计口径（某些损耗算正常、某些算异常）

规则要“可版本化”：同一工厂不同时间口径变了，要能保留历史版本，否则报表会对不上。

2.2 插件位：适合“需要对接设备/外部系统/复杂算法”的差异

插件不要到处开口子，只在**少数关键“钩子点（Hook Points）”**预留，做到可控。

建议预留的钩子点（最实用的 10 个）：

单据创建前（beforeCreate）：改默认值、补字段、编码

单据提交前（beforeSubmit）：复杂校验、合规检查

单据提交后（afterSubmit）：自动生成下游任务/通知

状态流转前/后（beforeTransition/afterTransition）：控制流程、联动库存/质检

成本结算前/后（beforeCosting/afterCosting）：企业口径差异

追溯计算前（beforeTrace）：不同追溯逻辑（箱→托→批等）

设备数据接入（scale/camera/iot adapter）：电子秤、扫码枪、相机

外部系统对接（erp/wms/财务）：字段映射、同步策略

导入导出（import/export transformer）：客户奇葩 Excel 模板

消息通知（webhook）：钉钉/企微/短信/邮箱

工程要求（很重要）：插件必须权限隔离、资源隔离、失败可降级，不能让一个插件把主流程拖死。
